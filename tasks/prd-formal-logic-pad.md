# PRD: 論理学形式証明パズル (Formal Logic Pad)

## はじめに

論理学の形式証明をパズル形式で組み立てながら学べるウェブサイトを構築します。ユーザーはドラッグ&ドロップで命題を繋ぎ、Hilbert流の公理系（S, K公理）と推論規則（MP: Modus Ponens）を組み合わせて証明を構築できます。たとえば「SKからIを証明する」（φ→φの導出）といった体験ができます。

対象ユーザーは論理学初学者であり、数学的背景がなくてもパズルとして楽しみながら形式証明の基礎を学べることを目指します。

**参考**: [京都大学 高崎研究室 論理学教材](https://www2.yukawa.kyoto-u.ac.jp/~kanehisa.takasaki/edu/logic/logic6.html)

## 設計原則: 独立性

各ライブラリは**完全に独立**しており、相互に依存しない。別々に開発・テスト・完成できる。

| ライブラリ     | パス                       | 依存先                  | 備考                                |
| -------------- | -------------------------- | ----------------------- | ----------------------------------- |
| InfiniteCanvas | `src/lib/infinite-canvas/` | React のみ              | 論理学の知識なし。汎用D&Dキャンバス |
| Logic Core     | `src/lib/logic-core/`      | Effect.ts のみ          | UIなし。完全ピュア。完全テスト      |
| Logic Lang     | `src/lib/logic-lang/`      | Logic Core の型定義のみ | 独立したlexer/parser。UIなし        |

- InfiniteCanvas は論理学と**一切無関係**な汎用ライブラリ
- Logic Core はUI・言語パーサーと**無関係**なピュアロジック
- Logic Lang は独立したDSL。Logic Coreの型を出力先として使うが、パーサー自体は独立

## 目標

- 論理学初学者がパズル感覚で形式証明を学べる環境を提供する

- 命題論理（Hilbert系: S, K公理 + MP）をまず実装し、後に述語論理も対応する

- 論理式スキーマ記述言語を**独立したライブラリ**として設計
- CIを通る状態に修復し、品質を維持する

#### US-009: 論理式スキーマASTの定義 [x]

**説明:** 開発者として、論理式スキーマを型安全なASTで表現したい。Effect.ts SchemaのTaggedClassを活用する。

**受け入れ基準:**

- [x] `src/lib/logic-core/`にピュアなロジックライブラリを構築
- [x] Effect.ts Schemaの`Schema.Class`でタグ付きクラスを定義
- [x] 以下の**論理式（Formula）**ASTノードを定義:
  - `MetaVariable`: メタ変数（φ, ψ, χ, ...）
  - `Negation`: 否定（¬φ）
  - `Implication`: 含意（φ→ψ）
  - `Conjunction`: 連言（φ∧ψ）
  - `Disjunction`: 選言（φ∨ψ）
  - `Universal`: 全称量化（∀x.φ）
  - `Existential`: 存在量化（∃x.φ）
  - `Predicate`: 述語適用（P(t1, t2, ...)）

- [x] 以下の**項（Term）**ASTノードを定義:
  - `TermVariable`: 項変数（x, y, z, ...）
  - `TermMetaVariable`: 項メタ変数（述語論理でのメタ項）
  - `FunctionApplication`: 関数適用（f(t1, t2, ...)）
  - `BinaryOperation`: 二項演算（t1 + t2, t1 \* t2, ...）
    - 演算子: `+`, `-`, `*`, `/`, `^` など（意味は与えない、構文のみ）
  - `Constant`: 定数（0, 1, a, b, ...）
- [x] discriminated unionとして`Formula`型と`Term`型を定義
- [x] **等号付き論理（equality logic）はオプションとして選択可能**
  - 体系設定で等号公理を含むかどうかを指定（Equality ASTノードは常に含む。バリデーション段階で制御）
- [x] すべてのフィールドはreadonly

#### US-016: 言語のパーサー [x]

**説明:** 開発者として、トークン列をASTに変換するパーサーを実装したい。

**受け入れ基準:**

- [x] 再帰下降パーサーまたはPratt parserで実装
- [x] **論理式**の演算子優先順位と結合性を正しく処理
  - 優先順位（高→低）: `~` > `/\` > `\/` > `->` > `<->`
  - `->` は右結合、`<->` は右結合または非結合
- [x] **項**の演算子優先順位と結合性を正しく処理
  - 優先順位（高→低）: `^` > `*`, `/` > `+`, `-`
  - `^` は右結合、他は左結合
- [x] **等号**は項と項を繋ぐ（論理式を生成）
  - `t1 = t2` は `Equality(t1, t2)` を生成
- [x] 関数適用: `f(t1, t2, ...)` は `Function("f", [t1, t2, ...])` を生成
- [x] 述語適用: `P(t1, t2, ...)` は `Predicate("P", [t1, t2, ...])` を生成
  - 識別子が大文字で始まれば述語、小文字なら関数として区別（慣例）
- [x] 量化子のスコープ: `all x. P(x) -> Q(x)` は `all x. (P(x) -> Q(x))`
- [x] パースエラー時にエラー位置と期待されるトークンを提示
- [x] ユニットテストで様々な式を検証:
  - `φ→φ`

## 機能要件

### キャンバスライブラリ

- FR-1: `<InfiniteCanvas>`は半無限空間を表示し、パン・ズーム操作が可能
- FR-2: `<CanvasItem>`はドラッグ可能なノードを表現
- FR-3: `<Connection>`は2つのアイテム間をベジェ曲線で接続
- FR-4: アイテムはコンテキストメニューを持てる

### 論理学コアライブラリ（ピュア）

- FR-5: 論理式スキーマ（Formula）と項スキーマ（Term）をEffect.ts Schema TaggedClassでAST表現
  - 項: 変数、定数、関数適用、二項演算（+, -, \*, /, ^）
  - 論理式: 述語、等号（オプション）、論理演算、量化
- FR-6: メタ変数（ギリシャ文字 + 添字0-999）を体系的に定義
- FR-7: メタ変数への代入を型安全に実行（変数捕獲チェック付き）
- FR-8: スキーマ間のユニフィケーション（代入推論）
- FR-9: 推論規則（MP、代入、公理導入、等号公理）を型安全に定義
  - 等号公理はオプション（体系設定で有効/無効を制御）
- FR-10: 証明図（木構造）の表現と検証

### 論理式スキーマ記述言語

- FR-11: DSLのトークン定義とLexer（位置情報付き、等号・二項演算含む）
- FR-12: 再帰下降/Pratt parserによるAST生成（項と論理式の優先順位処理）
- FR-13: Unicodeフォーマッター（→, ∧, ∨, ¬, ∀, ∃, =, +, −, ×）
- FR-14: LaTeXフォーマッター（KaTeX互換、等号・二項演算含む）

### ノートUI

- FR-15: ノートを開いて証明ワークスペースを表示
- FR-16: 公理をキャンバスに追加
- FR-17: ノード間を接続してMPを適用
- FR-18: 目標式の達成を判定

## 非目標

- 自然演繹やシーケント計算など他の証明体系はサポートしない（初期リリース）
- 自動証明器は実装しない（ユーザーが手動で構築する体験を重視）
- マルチユーザー・リアルタイムコラボレーションはサポートしない
- サーバーサイドでの証明保存は初期リリースではなし（ローカルストレージのみ）
- チュートリアル・ガイド機能は後続リリース

## デザイン考慮事項

- **ミニマル**: 論理学初学者が圧倒されないシンプルなUI
- **フィードバック**: ドラッグ中、接続時、証明完了時に明確な視覚フィードバック
- **ダークモード**: 将来的に対応を検討
- **レスポンシブ**: タブレットでも使用可能（タッチ操作対応）

## 技術的考慮事項

- **フレームワーク**: Next.js 16 + React 19（既存構成を継続）
- **Effect.ts**: 論理学コアライブラリで使用
  - `Schema.Class`でタグ付きクラスを定義（discriminated union）
  - 純粋関数的なAPIで副作用を分離
- **Storybook**: コンポーネント駆動開発、ライブラリの使用例を文書化
- **テスト**: Vitest + Testing Library、ロジック部分は単体テスト中心
  - 論理学ライブラリは完全なテストカバレッジを目指す
  - コーナーケース（変数捕獲、添字の区別など）を重点的にテスト
- **状態管理**: React状態 + useReducerを基本、必要に応じて検討
- **キャンバス描画**: SVG推奨（DOMベースでアクセシビリティに配慮）
- **パフォーマンス**: 大量ノード時の仮想化は後続で検討

## 成功指標

- ユーザーが「p→p」の証明を5分以内に完成できる
- ドラッグ&ドロップ操作が直感的（ユーザーテストで80%以上が迷わず操作）
- CIが常にグリーンで、PRごとにテストが通る
- Storybookでライブラリの全機能がデモ可能

## 未解決の質問

- キャンバスの永続化（ローカルストレージ vs IndexedDB）
- 式の入力方法（テキストボックス vs ビジュアルエディタ）
- モバイルでの細かな操作性（ピンチズームと接続線の競合）
- 公理選択UIの最適な形式（ドラッグ元パレット vs ダブルクリック追加）
