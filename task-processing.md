## 問題の調査結果

### 根本原因

`goalCheckLogic.ts` の `checkGoal()` で、SC/TAB系のスタンドアロンノード検証がスキップされている。

1. SC系は `extractLogicSystem()` → `emptyLogicSystem` を返す
2. `emptyLogicSystem` は `propositionalAxioms.size === 0` かつ `predicateLogic === false`
3. `hasAxiomSystem = false` → `doStandaloneCheck = false`
4. **結果: SC クエストでは、ゴール式と一致するノードが1つでもあれば、証明の完全性に関係なくゴール達成になる**

### 具体的な再現手順

1. SC クエスト sc-01 (ゴール: `phi -> phi`) を開始
2. ノードを作成し `⇒ phi → phi` と入力（空の前件 + ゴール式の後件）
3. **何も規則を適用しなくても** ゴール達成と判定される
4. `parseNodeFormula("⇒ phi → phi")` が正しく `phi → phi` を抽出し、ゴール式と一致

### Hilbert系との対比

Hilbert系では `doStandaloneCheck = true` になり:

- 孤立ノード（推論エッジに結論でも前提でも参加していない）は公理テンプレートに一致しない限りスキップ
- これにより「ゴール式を手動入力しただけ」のケースを排除

### 正しい挙動

SC で定理 `φ → φ` を証明するとは、シーケント `⇒ φ → φ` を導出すること。
導出は完全な証明木を構築して、すべての葉が公理（Identity: `φ ⇒ φ` または ⊥L）であること。

## 構造の比較検討

### 案 A: SC 用スタンドアロンチェック（推論エッジ接続必須）

SC/TAB でも `doStandaloneCheck` 相当のロジックを導入し、
ゴールにマッチするノードが「推論エッジに参加している」ことを要求する。

- **長所**: 最小限の変更。Hilbert と同じパターン
- **短所**: ルートノード（結論側として参加）は規則を1つ適用するだけで接続済みになり、
  葉まで完全に閉じているかは検証しない。部分的な証明でもゴール達成になりうる

### 案 B: 証明木の完全性チェック

ゴールにマッチするノードから証明木を辿り、すべての葉が公理であることを確認する。

- **長所**: 最も正確
- **短所**: 実装が複雑。`scTreeBuildLogic.ts` の既存ロジックを再利用する必要がある。
  ツリー構築失敗時のエラーハンドリングも必要

### 案 C: ルートノードの公理チェック（案 A の拡張）

SC でスタンドアロンチェックを有効にし、かつ SC の "axiom"（Identity, ⊥L）を
`isRecognizedAxiom` 相当でチェックする。

- **問題**: `isRecognizedAxiom` は Hilbert 公理テンプレートのパターンマッチで、SC公理とは別概念

### 推奨: 案 A（最小変更）→ 将来的に案 B

**理由:**

- 案 A で「ノードを置くだけ」の自明なケースは防げる
- SC では規則適用が結論→前提の方向で進むため、ルートノードに1つ規則を適用すれば
  そのノードは推論エッジの結論として接続される
- 完全性チェック（案B）は将来タスクとして分離可能
- 現行の Hilbert 系も完全な証明木の検証はしていない（孤立ノードの排除のみ）

### 実装方針

`goalCheckLogic.ts` の `checkGoal()` を修正:

1. SC/TAB/ND/AT 系でも `doStandaloneCheck` を有効にする
2. ただし `isRecognizedAxiom` は Hilbert 公理のため、SC 系では常に false
3. SC 系の孤立ノードは一律スキップ（= 推論エッジに参加していないノードはゴール達成に使えない）

具体的には `hasAxiomSystem` の条件を変更するか、非Hilbert系でも別の条件で standalone check を有効化する。

## ユーザー疑問への回答: 模範解答の方向性

> これは `phi => phi` から `=> phi->phi` を作るんじゃないの？そもそも模範解答もすべて修正が必要なのではないか

SC の証明木は**結論（根）から前提（葉）に向かって構築**する。

- 根: `⇒ φ → φ` （証明したい定理）
- →R 適用: `⇒ φ → φ` の前提として `φ ⇒ φ` を生成
- Identity 適用: `φ ⇒ φ` を公理として閉じる（葉）

模範解答 sc-01 はこの方向で構築しており正しい:
```
step 0: sc-root " ⇒ φ → φ"       ← ルートノード（結論）
step 1: implication-right on #0    ← →R 適用 → 前提 "φ ⇒ φ" 生成
step 2: identity on #1             ← Identity 適用 → 葉を閉じる
```

**修正不要。** 問題はゴールチェックロジックにある。
